Sync engine — design notes

Edited 20m ago·2,041 words·9 backlinks #sync#local-first

Goal: multi-device sync that never surprises the person typing. The invariant everything else hangs off: the file on disk is always a valid document, and the [[operation log]] is always sufficient to rebuild it. Cloud is a relay, not an authority — same stance as [[Local-first software]].

The operation log is the truth

Every mutation appends to the log before it touches state. Entries are JSONL with a hash chain — each entry carries the hash of its predecessor, so a torn write is detectable at the exact line it happened. Replay from any prefix must converge on the same document; that property is the whole design, and it's what the property tests pin down.

In local-first software, the availability of another computer should never be a barrier to your work. The data lives on your device, and the network is optional. — Kleppmann, Wiggins, van Hardenberg & McGranaghan, “Local-first software” (2019)

Conflict cases

Concurrent edits to different blocks merge trivially. Same-block edits go through the CRDT path; the nasty case is a delete racing an edit. Current rule: deletion wins only if the editing device has acknowledged the delete's log position — otherwise the block is resurrected with an edited-after-delete marker and surfaced in the conflicts panel.

Clock skew is real

delphi and the laptop disagree by ~40s after suspend. Order by log position, never wall time — see [[Incident 2026-06-14]] for what happens otherwise.

Compaction: the log grows ~2MB/month under normal use. Compact everything below the sync horizon into a snapshot entry, but write a fence first so offline devices can detect that history they depend on was folded:

sync/compact.rsrust
pub fn compact_below(log: &mut OpLog, horizon: Seq) -> Result<()> {
    log.append(Entry::fence(horizon))?;   // offline peers see this…
    let snap = log.snapshot_through(horizon)?;
    log.replace_prefix(horizon, snap)     // …and request a fresh snapshot
}
