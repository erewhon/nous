## Confirmed findings (highest severity first)

### DL-01 — Independent CRDT histories drop an entire side's blocks (blocks stored as Y.Map LWW value, not a shared array)

- **Severity:** CRITICAL  ·  network-IO
- **Files:** src-tauri/src/sync/crdt/converter.rs:86-129, src-tauri/src/sync/crdt/converter.rs:374-381, src-tauri/src/sync/manager.rs:1827-1832, src-tauri/src/sync/manager.rs:1918-1922, src-tauri/src/sync/manager.rs:1834-1850
- **Scenario:** A page is created/edited by a writer that only touches .json with no local .crdt (daemon HTTP, MCP, CLI, plugins, actions, Emacs nous.el — all call storage.update_page which writes .json only). On sync, prepare_local_doc_for_sync sees no .crdt and builds a FRESH Yjs doc via from_editor_data (independent client-id/history). A remote .crdt for the same page id exists (another device edited it independently). sync_page_concurrent_inner then calls local_doc.apply_update(remote_data). Because from_editor_data stores the block list under root.insert(txn,'blocks',ArrayPrelim) — the 'blocks' key is a value in the 'page' Y.Map — the two docs have NO shared array identity, so the merge resolves by Y.Map last-writer-wins on the 'blocks' key: one side's ENTIRE block list silently replaces the other's. Reproduced over 200 trials with random client IDs: 0 merges preserved both sides; ~50% lost local, ~50% lost remote. The merged .json (losing one side) is written to disk AND pushed to remote, propagating loss to every device. This is the mechanism of the 2026-05-01 incident; the seeding fix only covers crdt_existed==true, not the new-page / .json-only-page case.
- **Existing guards:** prepare_local_doc_for_sync seeds .json-only edits INTO an existing .crdt and a regression test exists, but the guard requires crdt_existed==true. The no-.crdt path builds a fresh independent doc with zero protection.
- **Suggested fix:** Never merge two independently-rooted docs via apply_update. When no local .crdt exists but a remote one does, do NOT build a fresh doc from .json and apply_update; instead load the remote .crdt as the base (shared history), then seed local .json-only blocks into it via diff_blocks/apply_block_changes (the same path used for existing .crdt). Equivalently, in prepare_local_doc_for_sync, fetch remote first and use it as the CRDT base before seeding. Add an integration test that merges a fresh-from-json local doc with an independent remote doc and asserts BOTH sides' blocks survive.
- **Suggested test:** Build local PageDocument::from_editor_data([L1,L2]) (no .crdt on disk), build an independent remote PageDocument::from_editor_data([R1]), run the full sync_page_concurrent_inner path (or prepare+apply_update), assert merged result contains L1, L2, AND R1. Today it contains only one side.

### DL-02 — Native/file-based page writes (database rows, chat, markdown, calendar, jupyter, canvas) are non-atomic fs::write with no temp+rename, no fsync, and no oplog/snapshot backstop

- **Severity:** CRITICAL  ·  network-IO
- **Files:** src-tauri/src/storage/file_storage.rs:2518-2535, src-tauri/src/bin/cli/api.rs:3548-3550, src-tauri/src/bin/cli/api.rs:3641-3643, src-tauri/src/bin/cli/api.rs:3821-3823, src-tauri/src/bin/cli/api.rs:2941-2943, src-tauri/src/commands/file_pages.rs:195, src-tauri/src/plugins/api.rs:533, src-tauri/src/plugins/api.rs:644, src-tauri/src/plugins/api.rs:753
- **Scenario:** All file-based page mutations (Database/Markdown/Chat/Jupyter/Canvas/Calendar) do read-modify-write of the entire source file (e.g. {id}.database JSON) and persist via write_native_file_content, which does a single fs::write(&file_path, content) — truncate then write, NON-atomic. Reached on every edit via daemon HTTP row add/update/delete (api.rs), MCP add_database_rows/update_database_rows, plugin host writes, and the Tauri update_file_content command (file_pages.rs). If the process is killed, the disk hits ENOSPC, or the write is interrupted after truncation but before the full buffer is written, the file is left empty or mid-JSON. On next read serde_json::from_str fails and the ENTIRE database (all rows) is unreadable. Unlike standard pages, file-based pages get NO oplog entry and NO snapshot (those only exist in update_page for standard .json pages), so there is no historical copy anywhere — content is permanently lost.
- **Existing guards:** None for the native-file path. atomic_write (temp+rename) is deliberately NOT used here. oplog and snapshots are wired only into update_page for standard pages. The storage Mutex serializes intra-process writers but does not make the single fs::write atomic; a crash mid-write still corrupts.
- **Suggested fix:** Route write_native_file_content through the same temp-file + fsync + atomic rename mechanism as atomic_write (write a unique sibling temp, fsync it, rename over the target, fsync the parent dir), and add recover_tmp_file handling for these files on load. Critically, also give file-based pages (especially Database) an oplog/snapshot safety net (or a rolling backup) before each write, since a single corrupted DB file currently loses every row with no recovery.
- **Suggested test:** Integration test: create a database page with N rows, then simulate a failed write (write to temp and inject error/kill before rename, or fill a tmpfs). Assert that after the failure the original file still parses and contains all N rows, and that two interleaved writes never leave invalid JSON on disk.

### DL-03 — No snapshot taken before destructive overwrite; snapshots are periodic (every 20th save) and capture POST-overwrite content, and oplog stores no content

- **Severity:** CRITICAL
- **Files:** src-tauri/src/storage/file_storage.rs:444-483, src-tauri/src/storage/snapshots.rs:38-83
- **Scenario:** update_page() does: read old content -> atomic_write(NEW content) -> append oplog -> THEN take_snapshot if should_snapshot. take_snapshot writes the page's CURRENT (already-overwritten) content, and should_snapshot only fires when oplog_count >= last_snapshot_count + 20. So a snapshot is never taken before overwriting, and on most saves none is taken at all. Concrete loss: a user (or AI agent / sync apply / external editor) replaces 500 words with one paragraph. The old text is gone from .json. The oplog only stored block-level diffs (BlockOp::Modify with block_id, no content). If the last snapshot was a few saves ago and lacks the deleted text, the original is unrecoverable. The headline guarantee 'operation log so loss is recoverable' does not hold because the oplog stores only hashes and change kinds, not content.
- **Existing guards:** atomic_write prevents torn writes of the .json itself; periodic snapshots exist but capture post-edit state and only every 20 saves; oplog records block-level change kinds but no block content.
- **Suggested fix:** Take a snapshot of the OLD content (read at the top of update_page) BEFORE atomic_write whenever the content hash changes and no recent snapshot covers the pre-edit hash. Alternatively store full pre-edit block content (or a content-addressed blob) in the oplog entry so any historical state is reconstructable. At minimum force a snapshot on the first edit after load and on large deletions (block_count drop > N or content length drop > X%).
- **Suggested test:** Rust test: create a page with rich content, call update_page 3 times with progressively smaller content (never reaching the 20-save threshold), then assert the original content is recoverable from snapshots+oplog. With current code this fails.

### DL-04 — Cross-process clobber: desktop DB editor and MCP/daemon DB-row writes (and stale update_page) silently overwrite each other via full-file last-writer-wins with no lock or version check

- **Severity:** CRITICAL  ·  network-IO
- **Files:** src/components/Database/DatabaseEditor.tsx:128-156, src/components/Database/DatabaseEditor.tsx:185-209, src-tauri/src/commands/file_pages.rs:194-198, src-tauri/src/bin/cli/api.rs:3497-3550, src-tauri/src/bin/cli/api.rs:1485-1640, src-tauri/src/bin/cli/api.rs:1688-1719, src-tauri/src/sync/crdt/store.rs:162-233, src-tauri/src/sync/crdt/store.rs:176, src-tauri/src/bin/cli/daemon.rs:128, src-tauri/src/commands/page.rs:113-143
- **Scenario:** The Tauri desktop app and the daemon are SEPARATE processes, each guarding its own FileStorage with its own in-process Mutex; there is NO cross-process flock or optimistic-concurrency check on page.json or .database content (page.updated_at is overwritten, not compared). Two concrete loss vectors: (1) The desktop DatabaseEditor loads the entire .database into memory and on a cell edit schedules an 800ms-debounced save that serializes the WHOLE snapshot and writes it back via write_native_file_content. Meanwhile an MCP agent adds a row through the daemon. Sequence: desktop loads [A,B]; MCP adds C -> file [A,B,C]; user edits cell in A; 800ms later desktop writes [A',B], silently deleting MCP row C. The only mitigation is a best-effort 'mcp-database-updated' WS event that can arrive after the debounce fires or be missed during reconnect. (2) For standard pages, the daemon's CRDT merge only protects a page that is live in the daemon's CrdtStore (requires a desktop pane_open WS frame). An MCP update_page without a pane_id uses pane='default'; if unregistered, apply_save falls back to current CRDT state or returns None and the daemon does a blind page.content = content full overwrite — a stale MCP snapshot deletes blocks added in the interim. append_to_page reads-then-full-writes with no CRDT routing at all. This is the exact divergence class behind the 2026-05-01 wipe.
- **Existing guards:** Per-process Mutex only; no cross-process file lock (only the Tantivy writer lock is cross-process-guarded). Best-effort 'mcp-database-updated' WS event triggers refreshFromDisk but is async/lossy. CRDT apply_save merges correctly ONLY when the page is live in the daemon's CrdtStore. No version/ETag/If-Match on HTTP writes.
- **Suggested fix:** Make the daemon the single writer of page.json and .database content (route all desktop writes through the daemon HTTP API), OR add optimistic concurrency: require an expected updated_at/content-hash precondition and return 409 on mismatch so the desktop must refresh+merge. Combine with a cross-process advisory lock (flock) around each read-modify-write. For standard pages, always seed the CRDT from the on-disk .json and apply a block-diff against current content instead of a blind page.content = content when apply_save returns None. Have the desktop DB editor PUT row-level deltas via the daemon's add/update/delete row endpoints rather than overwriting the whole file from a stale snapshot.
- **Suggested test:** Integration test: load DB content into an in-memory editor snapshot; via the daemon path add a row; then call Tauri update_file_content with the stale snapshot; assert the daemon-added row still exists on disk. Separate test: page on disk has [A,B,C]; simulate MCP update_page with stale [A,B] (no pane, not live); assert block C is preserved. Also spawn two FileStorage instances over the same temp dir, both update_page the same page id, and assert the second write is rejected (version check) or both edits are detectably preserved.

### DL-06 — Asset pull streams directly into the destination file (File::create truncates first); interrupted download corrupts/truncates the local asset and destroys the previous good copy

- **Severity:** HIGH  ·  network-IO
- **Files:** src-tauri/src/sync/webdav/client.rs:412-465, src-tauri/src/sync/webdav/client.rs:454-462, src-tauri/src/sync/manager.rs:2380, src-tauri/src/sync/manager.rs:2606
- **Scenario:** get_to_file does tokio::fs::File::create(local_path) — which TRUNCATES any existing file at the final path to zero — then streams HTTP body chunks one by one. If the connection drops, the server resets, a TLS error occurs, or the 300s timeout fires mid-body, the function returns Err but has already left a partially-written, truncated asset at the real path; the previous complete local copy is gone. There is no temp-file + atomic rename and no Content-Length verification. For an asset that existed and was locally modified (newer than remote), the original bytes are irrecoverable. Both the legacy pull and the CAS pull (which overwrites local when hashes differ) have this exposure.
- **Existing guards:** None — direct streaming to the final path with no atomic rename and no rollback on partial write. Caller treats Err as failure and does not mark synced, but the on-disk file is already truncated.
- **Suggested fix:** Stream to a sibling temp file (local_path + '.part'), fsync, then std::fs::rename to the final path only after the stream completes and validates. If the response advertises Content-Length, compare it against bytes written and error (deleting the temp) on mismatch; for chunked responses, rename only on a clean stream end. On any stream error, delete the temp and leave the original intact.
- **Suggested test:** Mock a body stream (or a mock WebDAV server sending a Content-Length larger than the body it writes, then closing the socket) that yields one chunk then an error; call get_to_file against an existing local file with known content; assert get_to_file returns Err AND the pre-existing local file is unchanged (not truncated/zero-length).

### DL-07 — atomic_write does not fsync the temp file before rename nor fsync the parent directory after rename — power loss can leave a renamed-but-empty/stale file despite the 'atomic' rename

- **Severity:** HIGH  ·  network-IO
- **Files:** src-tauri/src/storage/file_storage.rs:93-98, src-tauri/src/storage/oplog.rs:208-221, src-tauri/src/storage/snapshots.rs:65, src-tauri/src/storage/snapshots.rs:77
- **Scenario:** atomic_write does fs::write(&tmp_path, content) then fs::rename(&tmp_path, path) with NO File::sync_all on the temp before rename and NO fsync of the parent directory after. On ext4 (default data=ordered, delayed allocation) the rename's directory entry can reach disk while the temp file's data blocks have not, leaving the target file pointing at a zero-length/partial inode after a crash — an empty/corrupt page even though the save 'succeeded' and the daemon returned 200. The parent dir is never fsynced so even the rename may not survive. Every standard page save, folders.json/sections.json save, and file-based metadata save goes through this; sync writes pages this way too. append_entry (oplog) and snapshot writes are likewise unsync'd, so a crash mid-append leaves a partial JSONL line and a snapshot .json can exist with no .meta.json.
- **Existing guards:** rename is atomic w.r.t. which inode the name points at (prevents a half-written name) and recover_tmp_file handles the crash-before-rename case, but neither guarantees the contents behind a renamed name are flushed. folders.json/sections.json have no snapshot at all. read_entries silently skips unparseable oplog lines.
- **Suggested fix:** In atomic_write: open the temp file, write_all, File::sync_all() before rename, then open the parent directory and sync_all() it so the rename is durable. Apply the same to write_native_file_content and .crdt writes. For append_entry: flush()+sync_all() after writeln (or O_DSYNC). For snapshots: write page json + meta to temp, fsync, rename both, fsync dir; write meta last so a snapshot without meta is treated as incomplete and ignored.
- **Suggested test:** Use a crash-injection harness (dm-flakey or an fsync-counting fake FS) to verify atomic_write issues fsync on the temp file and the parent dir; a property test asserting that after a simulated crash at any syscall boundary the target file is either fully old or fully new content. Unit-test that a snapshot dir with a .json but no matching .meta.json is not treated as restorable.

### DL-08 — atomic_write uses a fixed, non-unique temp filename — two concurrent writers (app + daemon, or two threads) interleave into the same .tmp and rename corrupt JSON into place

- **Severity:** HIGH  ·  network-IO
- **Files:** src-tauri/src/storage/file_storage.rs:93-98, src-tauri/src/lib.rs:170, src-tauri/src/lib.rs:352, src-tauri/src/bin/cli/daemon.rs:128
- **Scenario:** The temp path is path.with_extension('json.tmp') — a deterministic name shared by all writers of that page. The Tauri app and the standalone daemon are SEPARATE processes that both open the same library and write page files; the app only checks the daemon PID file to avoid duplicate sync schedulers, not concurrent page writes. Each process has its own Arc<Mutex<FileStorage>>, so the mutex serializes only within one process. If the app saves page X (writing {X}.json.tmp) while the daemon also saves page X (same {X}.json.tmp), the two fs::write calls interleave bytes, then a rename promotes a corrupt mixture into {X}.json; serde then fails to parse and the page is lost. Even within one process the best-effort old_content read at update_page is outside any rename barrier, so a racing reader can read a half-written file in the ENOSPC/native-path case.
- **Existing guards:** Within a single process Arc<Mutex<FileStorage>> serializes writes. There is NO cross-process file lock (no flock/lockfile around writes). recover_tmp_file cannot tell a corrupt interleaved temp from a good one.
- **Suggested fix:** Use a unique temp filename per write (PID + thread id + uuid suffix, e.g. {id}.json.<uuid>.tmp) so concurrent writers never share a temp file. For cross-process safety add an advisory lock (flock on the page file or a per-notebook lock file) around read-modify-write sequences, or designate a single writer (route all app writes through the daemon API). Update recover_tmp_file to match the new temp-name glob.
- **Suggested test:** Spawn two threads/processes both calling update_page on the same page id concurrently in a loop; assert the file always parses as valid JSON and equals one of the two writers' payloads (never byte-interleaved). Verify temp filenames are unique per writer.

### DL-09 — Encrypted page writes and create/import/bulk re-encrypt paths use raw fs::write (non-atomic) with no oplog/snapshot — crash mid-write destroys the only copy

- **Severity:** HIGH  ·  network-IO
- **Files:** src-tauri/src/storage/file_storage.rs:2759, src-tauri/src/storage/file_storage.rs:2785, src-tauri/src/storage/file_storage.rs:368, src-tauri/src/storage/file_storage.rs:383, src-tauri/src/storage/file_storage.rs:398, src-tauri/src/storage/file_storage.rs:2488, src-tauri/src/storage/file_storage.rs:2878, src-tauri/src/storage/file_storage.rs:2914, src-tauri/src/storage/file_storage.rs:2953, src-tauri/src/storage/file_storage.rs:634, src-tauri/src/storage/file_storage.rs:2649-2665, src-tauri/src/storage/file_storage.rs:2122-2130
- **Scenario:** update_page_encrypted writes the encrypted page with a plain fs::write — truncate then write, non-atomic. A crash/ENOSPC mid-write truncates the ciphertext; encrypted pages have no oplog/snapshot fallback (those exist only in plaintext update_page), so the content is unrecoverable AND a truncated ciphertext may fail to decrypt entirely (losing the whole page). The same non-atomic pattern is used in create_page, create_page_from, create_page_with_id (the sync remote-page materialization path), create_page_encrypted, import_file_as_page, the bulk encrypt_all_pages/decrypt_all_pages/reencrypt_all_pages loops, and move_page_to_notebook. The bulk re-encrypt loops are especially dangerous: a crash partway truncates a page mid-rewrite, destroying its only copy. Separately, a truncated page .json from create_page_with_id makes list_all_pages (serde_json::from_str with ?) return Err, aborting the whole notebook listing and all future syncs.
- **Existing guards:** None — these paths bypass atomic_write entirely; encrypted pages have no oplog/snapshot; recover_tmp_file does not apply because no .tmp is created. list_all_pages has no per-file error tolerance.
- **Suggested fix:** Route ALL page-file writes (encrypted writes, creates, imports, and the bulk encrypt/decrypt/reencrypt loops) through a durable atomic_write (unique temp + fsync + rename + dir fsync). For bulk re-encryption, write each page to a temp and rename so a crash never truncates an existing page; take a snapshot before re-encrypting each page. Make list_all_pages tolerant: log and skip a single unparseable file (attempt .tmp recovery) instead of failing the whole listing.
- **Suggested test:** Enable encryption, simulate a write failure during update_page_encrypted, assert the prior content is still decryptable. Run reencrypt_all_pages with an injected failure on page k and assert pages 0..k and k.. are all still individually decryptable. Also: write a truncated .json into a notebook pages dir and assert list_all_pages returns the other pages and sync does not abort.

### DL-10 — Read-only collab permission enforcement is bypassed after Durable Object hibernation — a read-only guest can overwrite/corrupt the shared CRDT

- **Severity:** HIGH  ·  network-IO
- **Files:** collab/server/src/index.ts:130, collab/server/src/index.ts:202, collab/server/src/index.ts:220-234
- **Scenario:** connectionPermissions is a module-level in-memory Map populated ONLY in onConnect(). With options.hibernate=true, after inactivity Cloudflare evicts the DO's JS instance while WebSockets stay open. When a hibernated read-only ('r') guest sends the next message, partyserver routes it through webSocketMessage -> #ensureInitialized (onStart/onLoad only) -> onMessage; onConnect is NOT re-invoked, so connectionPermissions is empty. In onMessage, permissions = connectionPermissions.get(connection.id) returns undefined, the `permissions === 'r'` checks are false, and the sync-update drop is skipped. The RO guest's document updates (sync type 0 sub-type 2) are applied to the shared Yjs doc and broadcast to all clients, overwriting/corrupting other collaborators' content. This is the normal lifecycle for any idle session.
- **Existing guards:** Read-only enforcement exists for live (non-hibernated) connections via the connectionPermissions Map and the sync-message drop in onMessage, but it does not survive hibernation. The Map is also process-global across all DO rooms (a correctness smell).
- **Suggested fix:** Do not rely on in-memory module state for security. Persist per-connection permissions in the WebSocket attachment (connection.serializeAttachment/deserializeAttachment, which survives hibernation) at onConnect time, and read them in onMessage via deserializeAttachment instead of the module-level Map. Alternatively re-derive permissions from the token stored in the attachment on each message.
- **Suggested test:** Integration test (miniflare/workerd): connect an RO client, apply a doc update (assert dropped), then force DO hibernation/eviction (or directly invoke webSocketMessage without onConnect to simulate wake), send another RO doc update, and assert the document is unchanged and the update was dropped.

### DL-11 — Collab server persists the Yjs doc only via a debounced timer — no flush on last-client disconnect or before hibernation, so trailing guest edits can be lost from durable storage

- **Severity:** HIGH  ·  network-IO
- **Files:** collab/server/src/index.ts:156-160, collab/server/src/index.ts:240
- **Scenario:** onSave (ctx.storage.put of the encoded Yjs state) is invoked only by a lodash.debounce handler on the doc 'update' event (2s debounce, 10s maxWait). There is no DO alarm scheduling a flush, and neither y-partyserver's onClose nor the override onClose calls onSave. If the last connected client edits then disconnects within the debounce window, or the DO is evicted/hibernated before the debounce fires, the most recent updates are never written to ctx.storage. Because SyncManager deliberately SKIPS pages with an active collab session (has_active_collab_session), and a guest-only page has no host-side editor writing local .json, the collab server's debounced ctx.storage is the ONLY durable copy of those guest edits — losing the trailing window loses real guest content with no backup.
- **Existing guards:** Yjs clients retain their own state and re-sync on reconnect, so loss is avoided IF a client with the latest state reconnects later. No server-side guarantee.
- **Suggested fix:** Flush on last-disconnect: override onClose to await this.onSave() when getConnections() becomes empty. Additionally schedule a DO alarm as a backstop so a pending debounced save is committed even if the JS instance is about to be evicted. Reduce reliance on the debounce-only path for final state.
- **Suggested test:** Worker test: connect one client, apply an update, immediately close the socket before 2s elapse, evict/re-create the DO, reconnect, and assert the previously-applied update is present in the restored document (i.e., onClose forced a flush).

### DL-12 — File-based page content (source_file, e.g. database rows) is never synced cross-device; merge/pull writes only metadata and source files live outside assets/, so DB row edits silently diverge/overwrite

- **Severity:** HIGH  ·  network-IO
- **Files:** src-tauri/src/sync/manager.rs:1935-1962, src-tauri/src/storage/file_storage.rs:2594-2604, src-tauri/src/storage/file_storage.rs:163-174, src-tauri/src/sync/manager.rs:2139-2176, src-tauri/src/sync/manager.rs:2427-2519
- **Scenario:** For non-Standard pages (Database, Markdown, Chat, Jupyter) the real content lives in a separate source_file (e.g. files/{id}.database) and page.content.blocks is metadata-only. The CRDT machinery operates exclusively on page.content: prepare_local_doc_for_sync seeds/diffs page.content, the merge derives merged_content = local_doc.to_editor_data() and persists it via update_page_metadata, which for a non-Standard page writes ONLY the .metadata.json and never the source_file. Meanwhile the source_file lives under files/ (not assets/), so neither sync_assets nor sync_assets_cas pushes/pulls it (both only discover files under assets/). Net effect: database/markdown body content edited on device A is not propagated to device B; a remote metadata pull can bump B's updated_at without B ever receiving A's row data, and a later B->A push of B's stale file overwrites A's newer content via last-writer-wins. A concrete cross-device data-loss vector for database row content.
- **Existing guards:** update_page_metadata correctly avoids clobbering source_file (writes metadata only) and the placeholder-pull guard avoids overwriting existing pages — but no mechanism actually syncs/merges source_file body content, so body divergence/loss is not prevented.
- **Suggested fix:** Include the notebook files/ directory (database/markdown source files) in the asset/CAS sync with conflict detection (content hash + ETag), or extend the page-sync to carry source_file content with a proper merge/conflict strategy. At minimum, detect divergence between local and remote source_file and surface a conflict instead of silently overwriting.
- **Suggested test:** End-to-end (or manager-level) test: create a Database page on device A with rows, sync; on device B add different rows, sync; assert both devices converge (or a conflict is recorded) and neither device's rows are dropped. Verify files/{id}.* is actually uploaded/downloaded.

### DL-13 — Transient manifest/changelog fetch or parse error treated as 'remote empty' → overwrites remote manifest/changelog and drops all state vectors

- **Severity:** HIGH  ·  network-IO
- **Files:** src-tauri/src/sync/manager.rs:955-956, src-tauri/src/sync/manager.rs:1305, src-tauri/src/sync/manager.rs:1352, src-tauri/src/sync/manager.rs:1296, src-tauri/src/sync/manager.rs:1497-1505, src-tauri/src/sync/manager.rs:1513-1516, src-tauri/src/sync/manager.rs:2447-2454
- **Scenario:** fetch_manifest_static uses client.get + serde_json::from_slice. A transient non-2xx (500/502/503/timeout) OR a JSON parse error (e.g. reading a manifest another client is mid-PUT, since remote PUTs are non-atomic, or a truncated/chunked body) returns Err. The code maps ANY Err identically to 'no manifest': manifest_existed=false and a fresh empty SyncManifest is created. Because !manifest_existed, the sync pushes this near-empty manifest back to remote, overwriting the real one — dropping page_state_vectors and PageSyncState for every page not synced this round. Loss of state vectors degrades/breaks future CRDT merges (forces independent-history comparisons → re-triggers DL-01) and breaks other clients' change detection.
